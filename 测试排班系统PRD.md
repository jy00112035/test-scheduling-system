## 1. 项目概述

### 1.1 项目背景

当前测试团队在人力安排上存在以下痛点：

- 测试需求与人力匹配依赖线下沟通，效率低下
- 新员工人力系数管理不规范，影响排期准确性
- 跨项目人力分配缺乏可视化工具，容易出现冲突
- 人力投入统计依赖人工整理，数据准确性差

### 1.2 产品目标

建设一套 Web 端测试排班系统，实现：

- 测试需求线上化提交与审批
- 可视化人力排布与冲突检测
- 新员工人力系数自动递增管理
- 人力投入数据自动化统计分析

### 1.3 目标用户

| 用户角色 | 用户描述 | 核心需求 |
|---------|---------|---------|
| 测试经理 | 负责提交测试需求，管理测试进度 | 快速提交测试需求，掌握测试人力投入情况 |
| 资源主管 | 负责整体测试人力排布 | 高效分配人力，避免冲突，优化资源利用率 |
| 项目经理 | 负责项目排期决策 | 在人力不足时调整测试周期 |
| 测试执行人员 | 实际执行测试的人员 | 清晰查看个人任务排期 |
| 字段管理员 | 配置系统基础数据 | 灵活配置版本类型等基础字段 |

### 2.2 业务主流程

#### 2.2.1 测试需求提交流程

```mermaid
flowchart TD
    A[测试经理/资源主管/项目经理登录] --&gt; B[填写测试需求表单]
    B --&gt; C{检查必填项}
    C --&gt;|缺失| D[提示补全信息]
    C --&gt;|完整| E[提交需求]
    E --&gt; F[通知资源主管]
    F --&gt; G[需求进入待排期状态]
```

**测试需求表单字段：**

| 字段名 | 类型 | 必填 | 说明 |
|-------|------|------|------|
| 产品信息 | 下拉选择 | 是 | 从产品列表中选择 |
| 版本号 | 文本输入 | 否 | 如 v1.0.0 |
| 测试开始日期 | 日期选择 | 是 | |
| 测试结束日期 | 日期选择 | 是 | 需晚于开始日期 |
| 人力需求 | 数字输入 | 是 | 单位：人/天 |
| 版本类型 | 下拉选择 | 是 | 可配置，如：功能测试、回归测试、紧急修复等 |
| 备注说明 | 多行文本 | 否 | |

#### 2.2.2 人力排布流程

```mermaid
flowchart TD
    A[资源主管/项目经理登录] --&gt; B[查看待排期需求列表]
    B --&gt; C[选择需求进入排布页面]
    C --&gt; D{选择排布方式}
    D --&gt;|AI 推荐| E[系统生成推荐方案]
    D --&gt;|人工排布| F[拖拽分配人力]
    E --&gt; G[人工调整]
    F --&gt; G
    G --&gt; H{冲突检测}
    H --&gt;|存在冲突| I[提示冲突信息]
    H --&gt;|无冲突| J[发布排期]
    J --&gt; K[通知相关人员]
```

**人力排布核心规则：**

1. **人力百分比分配：**
   - 一人天可分配到多个项目
   - 单日分配总和 ≤ 100%
   - 默认以 10% 为最小分配单位

2. **新员工人力系数管理：**
   - 人员入职时预设初始系数（如 0.3）
   - 按周自动递增（第 2 周 0.5，第 3 周 0.7，第 4 周 1.0）
   - 支持手动调整系数
   - 系统根据入职日期自动计算当前生效系数

3. **人力不足处理：**
   - 系统检测到人力缺口时，高亮提示
   - 资源主管可触发「项目经理协调」流程
   - 项目经理收到通知，可调整测试结束日期
   - 调整后需重新进行人力排布

#### 2.2.3 测试执行与关闭流程

```mermaid
flowchart TD
    A[排期发布] --&gt; B[测试执行人员查看任务看板]
    B --&gt; C[按排期向测试经理报道]
    C --&gt; D[执行测试任务]
    D --&gt; E{人力投入是否达标}
    E --&gt;|是| F[系统提醒测试经理关闭]
    E --&gt;|否| G[继续执行]
    F --&gt; H{是否需要补充人力}
    H --&gt;|否| I[关闭测试需求]
    H --&gt;|是| J[提交补充测试申请]
    J --&gt; K[项目经理审批]
    K --&gt;|同意| L[更新排期]
    K --&gt;|驳回| M[结束]
    K --&gt;|修改后同意| N[返回调整]
    N --&gt; J
```

**补充测试申请字段：**

| 字段名 | 类型 | 必填 | 说明 |
|-------|------|------|------|
| 产品信息 | 只读 | - | 关联原测试需求 |
| 原测试需求人力 | 只读 | - | 显示原始需求 |
| 当前已投入人力 | 只读 | - | 系统自动统计 |
| 补充测试原因 | 多行文本 | 是 | |
| 补充测试结束日期 | 日期选择 | 是 | |
| 补充测试人力需求 | 数字输入 | 是 | 单位：人/天 |

**项目经理审批操作：**
- 同意：直接通过
- 驳回：需填写驳回原因
- 修改后同意：可调整结束日期或人力需求，返回测试经理确认

### 2.3 核心功能模块

#### 2.3.1 人员管理模块

**功能描述：** 管理测试执行人员基础信息

**人员信息字段：**

| 字段名 | 类型 | 必填 | 说明 |
|-------|------|------|------|
| 姓名 | 文本 | 是 | |
| 工号 | 文本 | 是 | 唯一标识 |
| 入职日期 | 日期 | 是 | |
| 小组名称 | 下拉 | 是 | |
| 初始人力系数 | 数字 | 是 | 新员工入职时设定，如 0.3 |
| 系数递增规则 | 配置项 | 是 | 默认：每周 +0.2，4 周后达 1.0 |
| 当前人力系数 | 只读 | - | 系统自动计算 |
| 状态 | 下拉 | 是 | 在职/离职/休假 |

**系数计算逻辑：**

```
当前生效周数 = floor((当前日期 - 入职日期) / 7)

当前人力系数 = min(初始系数 + 每周递增值 * 生效周数, 1.0)
```

**配置项示例：**

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| 初始系数 | 0.3 | 新员工第 1 周系数 |
| 第 2 周系数 | 0.5 | |
| 第 3 周系数 | 0.7 | |
| 第 4 周及以后 | 1.0 | |

#### 2.3.2 任务看板模块

**功能描述：** 所有用户查看任务排期

**看板视图：**

| 列名 | 说明 |
|-----|------|
| 人名 | 显示测试执行人员姓名 |
| 当前人力系数 | 显示今日生效系数 |
| 小组名称 | 显示所属小组 |
| 周一 ~ 周日 | 显示本周每日排班，可切换周视图 |

**每日排班卡片信息：**
- 产品名称
- 版本号（如有）
- 投入人力百分比
- 测试经理姓名
- 版本类型标签

**交互功能：**
- 切换查看上周/本周/下周
- 自定义日期范围查看
- 点击卡片查看需求详情

#### 2.3.3 人力排布工作台

**功能描述：** 资源主管和项目经理进行人力分配的核心工作台

**左侧面板：待排期需求列表**
- 按优先级排序
- 显示需求基本信息
- 显示人力缺口提示

**右侧面板：人力排布视图**
- 行：测试执行人员（显示姓名、当前系数、小组）
- 列：日期（默认显示两周）
- 单元格：显示已分配任务

**交互功能：**
- 拖拽需求到指定人员和日期
- 点击单元格调整分配百分比
- AI 推荐按钮触发自动排布
- 冲突检测按钮检查冲突
- 一键发布排期

**冲突检测规则：**
- 同一人同一天分配总和 &gt; 100%
- 分配时人力系数 &gt; 当前实际系数
- 测试结束日期早于开始日期

#### 2.3.4 仪表盘模块

**功能描述：** 管理层查看实时数据概览

**统计指标：**

1. **当前并行测试产品数量**
   - 实时数量
   - 环比上周变化

2. **今日人力概览**
   - 测试执行总人数
   - 已投入测试人数
   - 空闲人数
   - 人力利用率（投入人数 / 总人数）

3. **版本类型平均投入人力**
   - 按版本类型分组
   - 仅统计已完成的测试需求
   - 计算平均人力投入（人/天）

**图表展示：**
- 并行测试数量趋势图（近 30 天）
- 人力利用率走势图（近 7 天）
- 各版本类型投入人力分布饼图

#### 2.3.5 统计报表模块

**功能描述：** 按产品维度统计人力需求与实际使用

**报表维度：**

| 维度 | 说明 |
|-----|------|
| 产品 | 按产品名称分组 |
| 时间周期 | 月度/季度/年度 |
| 版本类型 | 可选过滤条件 |

**报表字段：**

| 字段 | 说明 |
|-----|------|
| 产品名称 | |
| 测试需求数 | 周期内提交的需求数量 |
| 计划人力总需求 | 所有需求计划人力之和 |
| 实际人力总投入 | 实际排班人力之和 |
| 人力差异率 | (实际 - 计划) / 计划 |
| 平均测试周期 | 平均每个需求的测试天数 |

**导出功能：**
- 支持导出 Excel
- 支持导出 PDF

#### 2.3.6 基础配置模块

**功能描述：** 字段管理员配置系统基础数据

**可配置项：**

1. **版本类型配置**
   - 类型名称
   - 颜色标签
   - 启用/禁用状态

2. **产品信息配置**
   - 产品名称
   - 产品负责人
   - 所属业务线

3. **小组配置**
   - 小组名称
   - 小组负责人

4. **系数规则配置**
   - 周次与对应系数映射
   - 支持自定义递增周期

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 要求 |
|-----|------|
| 页面加载时间 | ≤ 2 秒 |
| 仪表盘数据刷新 | 支持实时刷新，间隔可配置 |
| 并发用户数 | 支持 100+ 并发用户 |
| 拖拽操作响应 | ≤ 100ms |
| AI 推荐生成时间 | ≤ 5 秒 |

### 3.2 安全需求

- 用户登录需支持企业 SSO
- 操作日志完整记录，可追溯
- 敏感数据加密存储
- 权限控制精确到按钮级

### 3.3 可用性需求

- 支持 Chrome、Firefox、Safari 最新版浏览器
- 响应式设计，支持 Pad 端访问
- 操作流程有明确的状态提示
- 错误信息友好易懂

### 3.4 可扩展性需求

- 预留 API 接口供后续集成
- 支持自定义报表字段
- 排班规则可配置化

## 4. 用户界面设计要点

### 4.1 人力排布工作台

- 采用甘特图类似的时间轴布局
- 拖拽操作有视觉反馈
- 冲突项红色高亮显示
- 人力缺口黄色预警

### 4.2 任务看板

- 日历视图清晰展示每日任务
- 不同版本类型用不同颜色区分
- 支持周视图和月视图切换

### 4.3 仪表盘

- 数据可视化采用卡片式布局
- 关键指标突出显示
- 图表支持钻取查看详情

### 5.1 功能验收

- [ ] 测试需求可正常提交、查看、关闭（支持不同角色提交）

- [ ] 仪表盘数据准确实时更新
- [ ] 统计报表可正确生成并导出

### 5.2 性能验收

- [ ] 页面加载时间 ≤ 2 秒

- [ ] AI 推荐生成时间 ≤ 5 秒

## 6. 风险与依赖

### 6.1 潜在风险

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| AI 推荐算法准确性不足 | 中 | 优先保证人工排布体验，AI 作为辅助 |
| 现有数据迁移复杂 | 中 | 提供数据导入模板，分批迁移 |
| 用户使用习惯改变阻力 | 高 | 提供培训文档，过渡期内双轨并行 |

### 6.2 外部依赖

- 企业 SSO 登录接口
- 产品信息基础数据（可从现有系统同步）
